3. 門店物料盤點Inventory.tsx

資料結構

欄位型別說明
onShelfstring架上數量（若有 bag_weight 則單位為克）
stockstring庫存數量（盒/袋/桶）
discardedstring倒掉數量

總計公式

若有 bag_weight（如花生、紅豆等散裝品）：
  bagValue = onShelf(克) / bag_weight
  總計 = bagValue + stock

若無 bag_weight：
  總計 = onShelf + stock

關聯品項（linkedInventoryIds）：
  linkedTotal = SUM(各子品項的總計)

到期日批次邏輯

1. StockEntryPanel 讓使用者輸入多筆 (到期日, 數量)
2. 自動加總 → 回寫 stock 欄位
3. 提交時合併同一 (product_id, expiry_date) 的數量
4. Upsert → 讀取舊值 → 刪除不在新列表的行 → 驗證筆數

提交流程（handleSubmit）

1. 驗證：所有非聚合品項的「庫存」欄必填（無則填 0）
2. 構建 session + items + frozenItems
3. submitWithOffline() 提交主表：
   - Upsert inventory_sessions (onConflict: 'id')
   - Upsert inventory_items (onConflict: 'session_id,product_id')
   - Delete 不在新列表的舊 items
4. 額外寫入（線上時）：
   - Upsert frozen_sales
   - Save supply tracker（其他區消耗品）
   - Upsert inventory_stock_entries（到期日批次）
5. Telegram 通知 + Audit Log

4. 前日用量計算公式

前日用量 = (D-1)庫存 + (D)叫貨 - (D)庫存 - (D)倒掉

其中：
  (D-1)庫存 = 前一天盤點的 bagValue + stock
      bagValue = bag_weight ? on_shelf / bag_weight : on_shelf
  (D)叫貨    = 該日 order_items 的 quantity
  (D)庫存    = 當日盤點的 bagValue + stock
  (D)倒掉    = 當日盤點的 discarded

特殊情況：
  - 若前一天沒盤點 → 顯示 "-"
  - 關聯品項：加總所有 linkedInventoryIds 的前日用量
  - 負值或異常值 → 顯示但不阻擋提交
Inventory.tsx 和 Order.tsx 都用同一套公式計算前日用量，但各自獨立查詢 DB。

5. 門店叫貨Order.tsx

頁面功能

1. 載入最新盤點庫存（跨樓層加總）
2. 載入建議叫貨量（suggestion.ts）
3. 使用者修改數量 + 備註
4. 提交叫貨

庫存查詢邏輯

1. 查 inventory_sessions WHERE store_id AND date < selectedDate，按日期倒序
2. 取最新有品項資料的那一天（可能多樓層）
3. 跨樓層加總：stock[pid] = SUM(onShelf / bag_weight + stock)

截止時間

deadline = 叫貨日期的隔天 08:00 台灣時間
例：4/7 叫貨 → deadline = 4/8 08:00+08:00 = 4/8 00:00 UTC
超過截止時間 → 無法修改（isPastDeadline）

提交流程

1. 構建 session（含 deadline）+ items（quantity > 0 的品項）
2. submitWithOffline() 提交：
   - Upsert order_sessions (onConflict: 'id')
   - Upsert order_items (onConflict: 'session_id,product_id')
   - Delete 舊品項
3. Telegram 通知

6. 叫貨建議算法suggestion.ts

Step 1：歷史資料載入（10 分鐘快取）

查詢範圍：2021-01-01 至今
資料來源：
  - inventory_sessions + inventory_items（跨樓層加總）
  - order_sessions + order_items
  - weather_records（溫度、降雨）
  - daily_revenue（營業額）

Step 2：每日用量計算

策略一（有連續盤點）：
  usage = (D-1)庫存 + (D)叫貨 - (D)庫存 - (D)倒掉

策略二（僅有叫貨資料）：
  usage ≈ 叫貨量（Excel 匯入歷史用）

Step 3：三階段匹配

階段條件最低天數基礎分
Tier 1 嚴格日類型 + 季節 + 溫差≤5°C + 降雨相同≥2天100 - 溫差×5
Tier 2 放寬日類型 + 季節 + 溫差≤8°C≥2天60 - 溫差×3
Tier 3 兜底同日類型+同季 → 同日類型 → 最近14天≥2天25~30
加分項：
  - 寒暑假匹配 +10 / 不匹配 +5
  - 營業額相似度 revenueBonus (0~15)
  - 近期權重：7天內×2.0, 14天×1.5, 30天×1.2

季節判斷：11-3月=cool, 4-10月=warm
寒暑假：7-8月, 1/20~2/15

Step 4：覆蓋天數

央廚休息日：週三(3) + 週日(0)

覆蓋日期 = [叫貨當天] + [接下來連續的央廚休息日]
例：
  週二叫貨 → 覆蓋 2 天（週二+週三休息）
  週六叫貨 → 覆蓋 2 天（週六+週日休息）
  週一叫貨 → 覆蓋 1 天

Step 5：最終建議量

1. matchDaysV2() 匹配歷史 → avgUsage（加權平均）
2. 覆蓋期間總需求 = SUM(各天估計用量)
3. 安全庫存缺口 = MAX(0, baseStock - currentStock)
4. 建議量 = MAX(totalDemand, safetyStockGap)
5. 四捨五入到 roundUnit（如豆花 0.5 單位）
6. 鮮食不扣庫存（每天需新鮮補貨）

7. 央廚各店叫貨總表OrderSummary.tsx

1. 查詢各店 order_sessions + order_items WHERE date = selectedDate
2. 按門店分組 → 按品項彙總
3. 顯示央廚成品庫存（product_stock_items 最新一筆）
4. 聚合品項庫存 = SUM(linkedInventoryIds 的 stock_qty)
央廚庫存為快照（非即時），若央廚當日未盤點，會顯示前一天的數據。

8. 央廚出貨Shipment.tsx

資料結構

欄位說明
order_qty門店叫貨量（從 order_items 讀取）
actual_qty實際出貨量（央廚填寫，可≠叫貨量）
received門店是否已收貨（初始 false）

提交流程

1. 選擇出貨人員
2. 叫貨品項：讀取 order_items → 填寫實際出貨量
3. 未叫貨品項：央廚可主動出貨（order_qty = 0）
4. Upsert shipment_sessions (onConflict: 'id')
5. Upsert shipment_items (onConflict: 'session_id,product_id')
6. Delete 不在新列表的舊品項
7. 可填寫央廚回覆（kitchen_reply）

9. 門店收貨確認Receive.tsx

1. 讀取 shipment_items WHERE session_id = "{storeId}_{date}"
2. 顯示叫貨量 vs 實際收到量
3. 門店填寫差異備註
4. 提交：
   - Update shipment_sessions SET received_at, received_by, receive_note
   - Update shipment_items SET received = true（全部品項）
5. Telegram 通知（有差異時）
收貨為全部一次確認，不支持部分品項分別確認。

10. 央廚成品庫存盤點ProductStock.tsx

1. 選擇盤點人員
2. 各成品填寫庫存量（支援到期日批次）
3. 提交：
   - Upsert product_stock_sessions (onConflict: 'id')
   - Upsert product_stock_items (onConflict: 'session_id,product_id')
   - Delete 舊品項
   - Upsert product_stock_entries（到期日批次）
4. 與門店盤點的到期日邏輯完全一致

DB 表

store_products (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  category TEXT NOT NULL,
  unit TEXT,
  shelf_life_days TEXT,
  base_stock TEXT,
  our_cost NUMERIC,
  franchise_price NUMERIC,
  visible_in TEXT DEFAULT 'both',
  linkable BOOLEAN DEFAULT false,
  linked_inventory_ids TEXT[],
  box_unit TEXT,
  box_ratio NUMERIC,
  bag_weight NUMERIC,
  sort_order INTEGER DEFAULT 0
)

關聯品項機制

Parent（聚合品項）：linkable=false + linkedInventoryIds=[子ID1, 子ID2...]
Child（子品項）：linkable=true

盤點時：
  - Child 各自填寫架上/庫存/倒掉
  - Parent 顯示 linkedTotal = SUM(各 Child 的總計)
  - Parent 自己的 stock 欄為唯讀

叫貨時：
  - 只叫 Parent（代表整個品類的需求）
  - 建議量基於 Parent 的前日用量（加總 Children）

12. 離線提交機制submitWithOffline.ts

流程：
1. 檢查 navigator.onLine
2. 若離線 → 存入 IndexedDB 佇列（yba_offline/pending_submissions）
3. 若線上 → 直接 Supabase upsert
4. 線上但網路中斷 → 自動降級為離線佇列
5. 網路復通 → OfflineBanner 提示手動同步

IndexedDB 結構：
  DB: yba_offline (version 1)
  Store: pending_submissions (keyPath: 'id')

  PendingSubmission = {
    id: "{type}_{sessionId}_{timestamp}",
    type: 'inventory' | 'order' | 'settlement',
    storeId, sessionId,
    payload: { session, items },
    createdAt: number
  }
submittingRef 鎖死風險：handleSubmit 用 useRef 防重複提交。若提交過程中頁面切到背景或網路超時，鎖可能不會釋放，導致按鈕永久無反應。需關閉 tab 重開才能解鎖。
